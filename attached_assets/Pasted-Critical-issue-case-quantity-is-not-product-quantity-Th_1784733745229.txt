Critical issue: case quantity is not product quantity

The current savings formula is described as:

(sourceUnitPrice − targetUnitPrice) × targetCaseSize

That appears to calculate savings for one target case, not the whole PO line. It also does not address what happens when the source and target case packs differ.

Example:

Source: 4 cases × 30 lb = 120 lb
Target: 4 cases × 20 lb = 80 lb

Moving the numeric quantity of 4 unchanged would under-order by 40 pounds.

The opposite case could create unwanted over-ordering.

Safest MVP rule

Only allow automatic routing when:

source effective pack quantity
=
target effective pack quantity

and the base units are compatible.

Effective pack quantity must include:

caseSize × innerPackSize

where applicable.

Different-pack alternatives can still appear in the dialog, but should show:

Different pack size — manual quantity review required.

Do not automatically route them during the initial M3B release.

Future different-pack calculation

When different-pack routing is added, preserve total base-unit demand:

required units =
source order quantity × source effective pack quantity

target case quantity =
round up(required units ÷ target effective pack quantity)

Then calculate:

projected savings =
source extended line cost − target extended line cost

The UI would also need to disclose any overage caused by rounding to full cases.

Confirm the savings baseline

Projected savings should compare against the price actually present on the source PO line—not necessarily the latest current value on the source vendor_item.

The proper comparison is generally:

Source quantity × source PO line case price
versus
Target quantity × target current case price

The PO line represents what the manager is currently about to buy.

If the source PO line has no reliable price, the system can fall back to the vendor-item price, but that should make the savings estimate less reliable.

At minimum, verify:

Quantity is included in the savings total.
Full effective pack geometry is used.
The source baseline comes from the PO line where possible.
The audit distinguishes savings per case from total projected line savings.

This should be checked before anything else.

#474 Concurrent routing race safety

Two simultaneous requests must not both route the same source line.

Use database-level protections such as:

Transactional row locks
A unique constraint on the original routed line
Atomic deletion or status transition
Idempotency record created inside the same transaction

Application-level checks alone are insufficient.

#477 Concurrent destination quantity safety

Two users may route products into the same destination PO at nearly the same time.

Quantity merging must use:

Row locking,
Atomic increments,
or an upsert backed by an appropriate unique constraint.

The system must never use:

read quantity
→ add quantity in application
→ write result

without locking, because one update can overwrite another.

#476 Destination PO compatibility

“Existing pending PO for that vendor” is too broad.

A destination PO should be compatible by at least:

Company
+ location
+ vendor
+ pending/draft status
+ required delivery or order date
+ purchasing cycle, if available

Also exclude a PO that:

Has been submitted
Has been received
Has been canceled
Has already been exported for upload, unless the user explicitly chooses it
Belongs to another location
Represents an old or unrelated order cycle

When multiple compatible pending POs exist, do not silently choose the first database result. Either:

Create a new draft PO, or
Ask the operator which PO should receive the lines.

For MVP, creating a new destination PO when the match is ambiguous is safer.

Verify target-vendor authorization

The recap explicitly mentions validation of source PO company ownership, but also confirm server-side validation that:

Target vendor belongs to the company
Target vendor is available to the selected location
Target vendor item belongs to that vendor
Target vendor item is active
Inventory item belongs to the company
The user has purchasing authority for the location

“Same inventory item” alone is not sufficient authorization.

Idempotency semantics

The current behavior for repeating the same request is good.

Also test:

First request:
Original line → Vendor A

Second request:
Same original line → Vendor B

That should return a clear conflict or the original routing result. It must not silently treat two materially different requests as equivalent.

The idempotency identity should include enough information to distinguish a retry from a new contradictory instruction.

#473 No source-price history

Define the behavior precisely.

Recommended:

Source PO line has a valid price, but no price history: allow routing; mark savings as unverified.
