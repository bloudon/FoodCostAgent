# Vendor Registry MVP Reset

## Decision

The vendor registry has not been used in production.

Do not proceed with the large migration, compatibility layer, reference-only classification project, manufacturer-role classification, complex diff process or preservation of the original 141-row seed.

Trim the implementation back to the MVP requirement.

## Actual MVP requirement

The registry exists to help a restaurant select a company it normally:

* Places orders with
* Receives invoices from
* Pays directly
* Downloads an order guide from
* Connects to through an ordering portal

It is not intended to contain every manufacturer, brand, buying group or company involved in the food supply chain.

## Delete the oversized seed

The current registry seed is development-only and has no production dependencies.

Remove:

* The 141-row hardcoded seed
* Manufacturer and brand records
* Reference-only records
* Unverified review records
* Complex migration-diff logic
* Compatibility logic designed solely to preserve unused seed rows
* Startup reseeding behavior

Replace it with a clean, versioned MVP seed.

No production-data migration is required.

## MVP keeper criteria

A vendor may be globally seeded only when at least one of the following is verified:

```text
The company directly accepts restaurant orders
The company invoices restaurant operators
The company operates a customer ordering portal
The company provides customer order guides
The company delivers directly to restaurant locations
The company supports a verified ordering connector or file workflow
```

## Minimal schema

Keep the current useful matching fields, but avoid adding research-heavy metadata.

```ts
platformVendorRegistry {
  id
  canonicalName
  normalizedName
  exactAliases[]
  aliases[]
  websiteDomains[]
  connectorId
  orderingUrl
  orderingMode
  countryCode
  serviceRegionCodes[]
  status
  source
  createdAt
}
```

### Required values

```text
status:
approved
pending
rejected
inactive
```

```text
source:
seed
user_submitted
global_admin
```

```text
orderingMode:
connector
portal_link
public_ecommerce
contact_vendor
```

Do not add for MVP:

* Vendor-role taxonomy
* Verification-status taxonomy
* Reference-only visibility system
* Parent-company relationship graph
* Industry memberships
* Research notes
* Acquisition history
* Usage analytics
* Exact delivery-area guarantees
* Manufacturer records

Parent-company relationships and additional classifications can be added later only when a demonstrated product need exists.

## Connector scope

Retain the existing working connectors.

Add shared connector definitions only where they provide immediate value:

```text
cut_and_dry
powernet_pnet
food_order_entry
```

Initial capabilities must remain conservative.

A portal link must not be described as punchout or cart population.

## Customer-created vendors

The global seed does not need to contain every regional or local vendor.

When a customer cannot find its vendor:

```text
Allow the customer to create the vendor immediately
Make the vendor usable by that company
Submit the vendor name and domain as pending
Do not block onboarding
```

Global administrators can later:

```text
Approve
Merge
Reject
Keep company-specific
Assign a connector
```

## Global growth rule

Promote a pending vendor globally when:

* Its website and identity are verified
* It directly accepts restaurant orders
* Multiple companies submit or use it
* A global administrator approves it

Do not conduct broad directory research merely to increase the registry count.

Real customer usage should determine expansion priority.

## Geography

Geography is optional ranking metadata.

Use it only when explicitly known:

```text
US-FL
US-GA
CA-ON
CA-QC
```

Do not treat missing geography as national coverage.

Do not guarantee that a vendor serves every ZIP within a listed region.

## Search priority

```text
1. Exact canonical-name match
2. Exact alias match
3. Website-domain match
4. Matching customer region
5. Matching customer country
6. Other approved vendors
```

## Implementation plan

### Ticket 1 — Remove the oversized development seed

* Delete the existing 141-row hardcoded seed.
* Remove continuous startup reseeding.
* Clear the development registry table.
* Confirm there are no production dependencies.

### Ticket 2 — Keep the minimal schema

* Preserve useful matching and connector fields.
* Add only `canonical_name`, `ordering_mode`, country and optional region codes if missing.
* Do not implement the larger research schema.

### Ticket 3 — Seed the MVP keeper list

* Insert only the verified direct-ordering vendors listed above.
* Preserve the seven existing connector-enabled vendors where appropriate.
* Use proper canonical names, aliases and domains.

### Ticket 4 — Add shared connector definitions

Add:

```text
cut_and_dry
powernet_pnet
food_order_entry
```

with conservative capabilities.

### Ticket 5 — Preserve the learning workflow

* Search the approved registry.
* Allow manual company vendor creation.
* Submit unknown vendors as pending.
* Allow global-admin approval, merge and rejection.

### Ticket 6 — Add lightweight geographic ranking

* Use country and region only as ranking boosts.
* Do not build delivery-area management.

## Definition of done

```text
The manufacturer and brand clutter is gone
The picker contains only actual ordering-vendor candidates
The seed is small and understandable
The registry recognizes common distributors
Known connectors are suggested
Customers can add any missing local vendor
Unknown vendors enter the admin-review workflow
No startup process continually rewrites the registry
No unnecessary migration or compatibility architecture remains
```

## Guiding rule

Before adding a company to the global ordering-vendor registry, ask:

> Would a restaurant normally place an order with this company and receive an invoice from it?

If the answer is no, do not seed it.
